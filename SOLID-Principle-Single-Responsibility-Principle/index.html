    Solid Principle Single Responsibility Principle

    2 minute read

    Single Responsibility Principle is a very easy but also confusing principle, that we may easily thought we’ve got it but actually we only did partially. We may have interpreted it as:

    Each Module should just do one thing.

    I have to confess, that’s how I understood it at the beginning. There is nothing wrong with this understanding, in fact it complies with the concept of designing modules to be Highly Conhesive, and Loosely Coupled. However, it’s hard to understand why is this principle important, and what happens if you violate it.

    Here comes the other definition of SRP which is popular on the internet, but doesn’t make much sense when I read it.

    Each module should have one, and only one, reason to change.

    What does this definition has any relationship with the word “Responsiblity”? At least for me, I don’t find a clue. So try another one:

    Each module should be responsible to one, and only one, actor.

    Actor is a group of users or stakeholders who requires this module, and demand changes in the same way.

    Now, I feel that’s much better to understand and articulate. I got this definition from Robert Martin’s book “Clean Architecture”, and I’m going to use his example in the book to illustrate this principle.

    Example of SRP Violation:

    public class Employee {
  public void calcuatePay() {
    // Financial Department use this function to calculate pay.
    ...
    getRegularHours() // To calculate the pay
    ...
  }
  
  public void reportHours() {
    // HR Department require this
    ...
    getRegularHours() // To report hours
    ...
  }
  
  private int getRegularHours() {
    // Based on Don't Repeat Yourself principle, the method here is a shared private function
    // used by both calcuatePay() and reportHours(), starting with a simple logic, 
    // say, calculate all the regular hours of regular work days.
    
    // Now here is the tricky situation:
    // What if the HR Department wants to change how this is calculated,
    // while Financial Department want to stay the same?
    
    // In real life, this is what happens:
    // Developer from HR Department makes the change, and test everything with HR's signoff.
    // After several days, the Financial Department discovered this change, because
    // They are overpaying millions of dollars...
  }
  
  public void save() {
    // A common method to persist this record, you can say DBA requires this.
  }
}

    As illustrated above, if there are multiple actors depending on a module, they could be affected by one single change without even knowing it. This is an example of deeply coupled code. This makes code change or refactoring extremely risky and hard, and normally people just lose the courage to do so.

    The fix is to extract the function for each actor and make a module for itself. And the developer from each department only needs to change the module they are responsible, without unnecessary code conflict or risky knockon effect.

    I’d like to end this post with the responsibility of a good architect:

    • Make the system easy to change and maintain.
    • Increase the productivity of each developer
    • Reduce the overall cost by achieveing the above two.
